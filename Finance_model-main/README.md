# Finance_model

# Кратко опишите предметную область: что моделирует программа
Программа моделирует упрощенную финансовую систему — «Учет финансов» ТигрБанка:
Счет (BankAccount): Место хранения денег пользователя
Категория (Category): Группирует операции по типу (Доход или Расход)
Операция (Operation): Фиксирует факт движения денег
Программа предоставляет консольный интерфейс для просмотра за сущностями и их аналитикой.

# Соблюдайте принципы SOLID
1. Single Responsibility Principle - Каждый класс имеет только одну причину для изменения
2. Open/Closed Principle - Классы открыты для расширения, но закрыты для модификации:
3. Liskov Substitution Principle - Все реализации интерфейсов взаимозаменяемы и сохраняют контракты
4. Interface Segregation Principle - Клиенты не должны зависеть от интерфейсов, которые они не используют:
5. Dependency Inversion Principle - Зависимости должны быть направлены на абстракции, а не на конкретные реализации

# Приведите аргументы, почему введенные абстракции улучшили качество дизайна
Высокоуровневые модули перестали зависеть от низкоуровневых
Зависимости направлены на абстракции, а не на конкретные реализации

Изменение в одном классе не требует изменений в классах-потребителях
Все зависимости настраиваются в одном месте (контейнер)

Классы занимаются только своей прямой задачей
Создание и управление зависимостями вынесено в отдельный механизм

Легко добавлять новые сервисы, не меняя существующий код
Просто расширять функциональность через новые реализации

# Укажите ситуации, при которых появятся проблемы добавления нового функционала или расширении бизнес-логики
## Добавление поддержки нескольких валют
При добавлении валют потребуется:
Добавить поле currency в BankAccount, Category, Operation
Изменить логику проверки баланса
Модифицировать все методы performTransaction/deposit/withdraw для работы с курсами валют

## Добавление периодических (регулярных) операций
Для добавления ежемесячных платежей потребуется:
Создавать новый тип сущности RecurringOperation
